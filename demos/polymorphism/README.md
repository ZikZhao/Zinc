### 实体规则与状态机设计

基于前面定义的接口，每个实体的行为逻辑（`tick` 和 `interact`）需要强制触发不同深度的调用栈和类型检查。网格更新采用 Moore 邻域（八向）。

1.  **VoidCell (空)**
    * 无主动行为。在被其他实体调用 `accept(IEntity)` 时，始终返回 `true` 并执行位置交换。
2.  **Wall (墙)**
    * 完全静态。拒绝一切物理位移请求（`accept` 返回 `false`）。
3.  **Sand (沙)**
    * **行为**: 每帧通过 `tryMove` 尝试向下（南）、左下（西南）、右下（东南）探测。
    * **交互**: 优先取代 `VoidCell`。如果下方是 `Water` 或 `Acid`，通过双重分派触发 `swap`（沙子下沉，流体上浮）。
4.  **Water (水)**
    * **行为**: 探测下、左下、右下、左、右。
    * **交互**: 遇到 `Lava` 触发激变；遇到 `Plant` 提供生长条件。
5.  **Plant (植物)**
    * **行为**: 静态。每帧检查邻域。
    * **交互**: 如果邻域内存在 `Water`，调用 `Water` 的 `consume()` 接口将其转化为 `VoidCell`，并以概率 $P_{grow}$ 寻找相邻的一个 `VoidCell` 生成新的 `Plant` 实例（堆分配新对象）。
6.  **Lava (岩浆)**
    * **行为**: 流体探测逻辑同 `Water`，但更新频率可通过内部的 `tick_counter` 降频（模拟高粘度）。
    * **交互**: 与 `Water` 接触时，将 `Water` 替换为 `Steam`，自身转化为 `Wall` 或 `Metal`（冷却）。
7.  **Steam (蒸汽)**
    * **行为**: 探测上、左上、右上。
    * **交互**: 存在内部生命周期变量 $\tau$。每帧 $\tau = \tau - 1$，当 $\tau \le 0$ 时，自身析构并原地生成 `Water`（凝结）。
8.  **Acid (酸)**
    * **行为**: 流体探测逻辑同 `Water`。
    * **交互**: 接触到 `Wall`、`Plant` 或 `Metal` 时，以概率 $P_{corrode}$ 销毁目标（替换为 `VoidCell`），并同时销毁自身（同归于尽）。
9.  **Metal (金属)**
    * **行为**: 静态抗腐蚀材质。拒绝所有流体的交换请求，降低 `Acid` 对其腐蚀的概率。
10. **Generator (生成器)**
    * **行为**: 初始化时绑定一个实体工厂函数的闭包/接口。每 $N$ 帧在相邻的空位上实例化其绑定的实体类型（例如 `WaterGenerator`, `LavaGenerator`）。
11. **BlackHole (黑洞)**
    * **行为**: 每帧遍历 Moore 邻域，调用所有非静态实体的 `destroy()` 接口，强制将其引用覆写为 `VoidCell`。

### 栅格冲突与并发更新解决策略

由于你的目标是压榨多态性能而非实现严格的物理并行计算，不建议使用带来巨大内存开销的 Double Buffering（双缓冲）。直接在单网格上进行**原地更新（In-place Update）**，这会带来读写冲突（例如两个沙粒同一帧想落入同一个空位）。

**解决方案：基于帧奇偶校验的乱序遍历与双重分派锁定**

1.  **防止重复更新 (Double Update):** 在 `IEntity` 基类中加入 `uint64_t last_update_frame`。主循环的当前帧序号为 `current_frame`。遍历到一个实体时，若 `last_update_frame == current_frame`，直接跳过。
2.  **空间冲突解决:** 不做全局仲裁。谁先在遍历循环中拿到 CPU 控制权，谁就直接执行内存覆写。
    * 当实体 A 决定移动到实体 B 的坐标时，A 调用 `B->acceptMove(A)`。
    * 如果 B 是 `VoidCell`，B 将自身引用写入 A 的原坐标，将 A 的引用写入 B 的原坐标，并返回 `true`。
    * 如果 B 拒绝（如 `Wall`），返回 `false`，A 留在原地。
3.  **消除方向性偏见 (Directional Bias):**
    * 每一帧交替改变网格的遍历顺序：从上到下/从下到上，从左到右/从右到左。这需要 4 种遍历策略在循环中轮换，能产生非常自然的流体散开效果。

### 动态平衡的数学抽象与维持机制

为了使系统达到永不宕机的混沌稳态（Dynamic Equilibrium），我们需要维持系统内的总物质（对象数量）处于一个动态区间。

定义系统状态空间向量 $\vec{S} = [n_{void}, n_{water}, n_{lava}, n_{steam}, \dots]^T$。
物质的增减由两组算子决定：
* **源算子 (Source $\Sigma$)**: `Generator` 每帧固定注入物质，$\frac{d}{dt} N_{matter} > 0$。
* **汇算子 (Sink $\Gamma$)**: `BlackHole` 无条件吞噬，以及 `Acid` 反应导致的物质湮灭，$\frac{d}{dt} N_{matter} < 0$。

**平衡条件构建：**
在地图顶部放置 `WaterGenerator`，底部放置 `LavaGenerator` 和少量 `BlackHole`。
1.  **反应链闭环**：`Water` + `Lava` $\rightarrow$ `Steam` + `Wall`。
2.  **相变闭环**：`Steam` 上升到达顶部，逐渐冷却凝结为 `Water`，再次下落。
3.  **死锁破除**：随着 `Wall` 不断生成，网格会被彻底堵死（生态僵死）。因此需要引入 `AcidGenerator`，定期从高空滴落酸液，腐蚀掉堆积的 `Wall`。

通过这种拓扑结构，网格内会不断发生“生成 - 阻塞 - 腐蚀坍塌 - 重新生成”的循环。堆积对象的生成与析构会持续触发内存分配器（Allocator）的压力，结合高频虚函数调用，形成极限 Benchmark 场景。

### 初始网格随机化算法 (针对 BTB 特性)

为了测试分支预测器，初始状态需要大面积的“同质区块”。随着模拟进行，流体和反应的交界处会产生“异质碎形”，导致 BTB 未命中率飙升。

**算法：多频柏林噪声阈值分割 (Multi-octave Perlin Noise Thresholding)**

1.  针对网格坐标 $(x, y)$，采样两个不同频率的二维 Perlin Noise：$N_{low}(x, y)$ 和 $N_{high}(x, y)$。
2.  使用 $N_{low}$ 划分宏观生物群系（Biome）：
    * $N_{low} < -0.3$: 分配为 `Water` 区域。
    * $-0.3 \le N_{low} < 0.2$: 分配为 `VoidCell` 区域。
    * $0.2 \le N_{low} < 0.6$: 分配为 `Sand` 区域。
    * $N_{low} \ge 0.6$: 分配为 `Wall` 区域。
3.  使用 $N_{high}$ 叠加微观细节和干扰：
    * 在 `VoidCell` 区域中，如果 $N_{high} > 0.8$，强制将其替换为 `Plant`。
    * 在 `Wall` 区域中，如果 $N_{high} < -0.7$，强制挖空为 `VoidCell`（形成天然洞穴）。
4.  在地图边缘和深处硬编码放置若干 `Generator` 和 `BlackHole`。

**BTB 效应预期：**
在第 0 帧，内存中的对象类型分布表现出极强的空间局部性。当 CPU 以行主序遍历网格调用 `ptr->tick()` 时，BTB 会完美预测跳转地址。
大约 $1000$ 帧后，随着重力沉降、对流混合、化学反应导致的新对象频繁生成，内存中的对象指针会被彻底洗牌。相邻网格的类型将变得完全不可预测（例如 $[Water, Steam, Void, Acid, Sand]$ 交错），此时虚函数的调用将退化为纯粹的间接跳转内存寻址（Indirect Call），你将能观察到各语言虚拟机/编译器在内联缓存（Inline Caching）和虚表（VTable）上的真实性能代差。





针对你提出的改进要求，我设计了一套基于“群系核心（Biome Cores）”和“受限生长（Constrained Growth）”的初始化随机算法。这套算法将不再进行全地图像素级的散布，而是分步生成：

1.  **确定群系核心：** 首先设定不同群系（墙体、沙地、植物、流体源、黑洞）的核心数量。将这些核心打乱顺序。
2.  **核心放置：** 依次在一个随机位置（满足特定地形条件）放置一个核心点。
3.  **群系扩张（关键）：**
    * **墙体脉络：** 采用一种“线性生长模型”，从核心点开始向前游走，在每前进一步后，有一定的概率发生大角度“转折”或“分支”。这使得墙体形成复杂的、网络状的脉络，而不是散乱的块。
    * **沙子聚集：** 使用一种“聚集扩散”模型（例如基于距离核心概率递减的DLA模型），形成一个庞大的、有边界的堆。
    * **植物森林：** 采用“扩散生长”模型，围绕核心和底部的墙/沙结构连片生长，形成有凝聚力的冠层或地下丛。
    * **流体：** 少量源点，然后通过物理模拟聚集。
4.  **整合：** 将所有生成的群系图层叠加在深灰色虚空背景上。

**视觉效果变化预示：**

在第一版效果图中，你可以看到实体是高度散乱的，几乎每个位置都有墙和绿植混杂。在改进版（即下一帧模拟或新开局）的预示中，你将看到：

* **上部：** 一个庞大的、复杂的、网络状灰色墙体结构横跨屏幕，有清晰的“折腾”路径。
* **中部：** 几个凝聚的大型黄色沙堆（包括金字塔）和绿植岛屿。
* **底部：** 一个庞大且更有结构的底层。
* **散布：** 绝大部分以前的单像素散乱点被移除，取而代之的是有凝聚力的、区域性的群系。你会看到少量的凝聚蓝色水池和极个别的特殊实体（如红色熔岩流）作为点缀，而不是散落一地。

这种更有凝聚力的结构将更真实地反映出分支预测器（BTB）在处理不同群系边界时的压力，因为相邻网格的类型将更加确定，但群系边界的突变将成为主要的预测失败源。





那个水平悬空的淡蓝色/灰色长条确实表现出了典型的静态材质特征。如果它是 `Metal`，由于它不受重力影响且拒绝流体的交换请求，形成这种悬空平台是符合规则的。它恰好可以作为测试传导接口（`IConductive`）绝佳的温床。

关于你的三个改进想法，从底层引擎和测试多态性能的角度来看，都非常有价值，我们可以逐一拆解其实现思路：

### 1. Lava Generator：打破热力学死寂
必须加。当前的截图显示沙子已经堆积成稳态，植物形成了固定的群系，如果没有强破坏性的实体介入，系统很快会进入一种“死寂态”。一旦大量实体处于静止（例如四面受阻，`tryMove` 瞬间返回），虚函数调用的实际开销会被掩盖。
加入 `Lava Generator`（作为强力 Source）配合底部的堆积物，能持续触发 `IReactive` 接口的动态分派，强制将 `Water` 转化为 `Steam`，将 `Plant` 焚毁。这种相态转换会不断析构旧对象并堆分配新对象，保持内存布局的碎片化和多态调用的高频活跃。

### 2. 游走生成器 (Kinematic Generators)：彻底摧毁分支预测
让 Generator 移动是一个极其精妙的决定。
* **BTB 的噩梦**：静态 Generator 会在固定区域生成同质实体，CPU 的分支目标缓冲区（BTB）很容易对这片区域的虚表跳转地址建立完美的预测模型。如果 Generator 开始游走，它就像一把刷子，在纯净的群系上强行画出极其复杂的异构交界线。
* **实现思路**：让 `Generator` 继承 `IMovable` 接口。为其内部增加一个速度向量 $\vec{v} = (v_x, v_y)$ 和一个内部帧计数器。在 `tick` 方法中，不再只是生成实体，而是先调用自身的 `tryMove`。
* **交互逻辑**：当移动的 Generator 撞击到 `Wall` 或其他不可跨越实体时，可以通过多态调用 `onCollide` 触发方向反转。这种动态的生成轨迹能确保整个网格内的类型分布时刻处于高熵状态。

### 3. 抑制流体高频抖动：引入状态惯性 (State Inertia)
目前流体在平地上的左右横跳（Jitter），是因为在传统的纯无状态元胞自动机中，向下受阻后的侧滑方向完全依赖于单帧的随机数（例如 `rand() % 2`）。这不仅视觉表现糟糕，而且高频的 PRNG 调用会污染 CPU 的执行端口。

要让流体表现得更平滑（具有趋向性），必须在 `Water` 或 `Acid` 等流体实例内部引入**惯性上下文**。

* **内存对齐与状态压缩**：为了不增加额外的内存开销（保持对象尽可能小以塞入更多进 L1/L2 Cache），可以利用对象内部的 Padding 或位域。比如用一个 `int8_t lateral_momentum`（取值 -1, 0, 1）来记录上一次的侧滑方向。
* **演化规则**：
    1.  当流体尝试向下移动失败时，检查 `lateral_momentum`。
    2.  如果 `lateral_momentum != 0`，优先尝试向该方向移动。
    3.  如果该惯性方向也受阻（例如撞墙），则将 `lateral_momentum` 翻转（-1 变 1，1 变 -1），或者概率性清零重置。
    4.  只有当 `lateral_momentum == 0` 时，才调用随机数发生器决定一个新的方向，并记录下来。

这种基于内部状态的“粘滞”侧滑逻辑，能让流体在遇到平地时，形成连贯的单向水流（类似于真实的铺展），直到遇到障碍物才发生回流。这不仅优化了视觉表现，还通过减少无意义的随机数调用，将 CPU 算力更纯粹地压榨在多态分派和内存解引用上。





引入热量累积与相变机制，以及物理游走型的黑洞，对压榨多态性能和内存分配器（Allocator）是极佳的策略。这会造成大量的局部对象析构与重建，使堆内存高度碎片化，进一步破坏空间局部性。

### 1. 热量累积与熔毁机制 (Heat Accumulation)

为了不增加单个对象的内存开销（保持对象尽可能小，以塞入更多进 L1/L2 Cache），可以利用 64 位架构下虚表指针（vptr）带来的内存对齐填充（Padding）。

* **状态压缩**：在 `Wall` 和 `Metal` 等静态实体中，增加一个 `uint8_t heat` 字段。这通常不会改变对象的 `sizeof`。
* **热力学传递**：
    * 在 `tick` 遍历时，若周围存在 `Lava`，通过接口查询或双重分派，触发当前实体的 `onHeat()` 方法，`heat` 增加。
    * 若周围没有热源，`heat` 按固定速率衰减。
* **相变阈值**：当 `heat > MELT_THRESHOLD` 时，实体内部逻辑触发自身的析构，并在当前坐标堆分配一个新的 `Lava` 实例（或者返回一个状态码交由外部网格管理器进行指针替换和回收）。
* **性能影响**：这种设计使得原本静态的顶部碎块区域变成了动态的“沸腾区”。大量的对象被高频 `delete` 和 `new`，会迅速打乱内存池的连续性，使得后续的虚表查找跨越更多的 Cache Line。

### 2. Metal 与 Wall 的物理/化学区分

在现实物理中，岩浆冷却后形成的是火成岩（如玄武岩、黑曜石），主要成分是硅酸盐，并非纯金属。因此，`Lava` 冷却转变为 `Wall`（代表石头）是符合逻辑的。金属（`Metal`）应该作为一种特殊的、由生成器生成或特定极端反应产出的稀有介质。

**在模拟中拉开两者的性质差异：**

* **传导与隔热**：`Wall` 是热绝缘体，只能被动累积热量直至熔毁。`Metal` 实现 `IConductive` 接口，它不仅自身累积热量，还会将热量**传递**给相邻的实体（例如将另一侧的水瞬间加热成蒸汽）。
* **抗腐蚀**：`Acid` 接触 `Wall` 时有较高概率将其销毁。接触 `Metal` 时概率极低。
* **冷凝催化（解决金属无存在感的问题）**：利用 `Metal` 极强的导热性，强制改变 `Steam` 的生命周期。当 `Steam` 的双重分派探测到邻居是 `Metal` 时，忽略其自身的生命周期 $\tau$，立刻强制析构并原地凝结为 `Water`。这会在金属表面形成明显的“挂水”和快速的相态折返，增加该区域的分支复杂度。

### 3. “DVD 屏幕保护”式黑洞 (Kinematic Black Hole)

将静态黑洞改为全图游走的动量实体，不仅视觉效果更混乱，而且它像一个“内存推土机”，会无视任何群系结构，在网格中强行碾出一条由 `VoidCell` 组成的空洞轨迹。

* **游走逻辑**：
    * 黑洞内部维护一个浮点型的位置坐标 $(x, y)$ 和方向向量 $(v_x, v_y)$。
    * 每帧计算新位置。如果检测到网格边界坐标，则对相应的速度分量进行反转（如触碰左边界，$v_x = -v_x$），实现完美弹射。
* **AoE 湮灭 (Area of Effect)**：
    * 设定半径 $R$（如 2 或 3）。
    * 将浮点坐标取整映射到网格索引。
    * 遍历局部窗口 $[x-R, x+R]$ 和 $[y-R, y+R]$。
    * 对窗口内的每一个网格索引 $(i, j)$，进行欧氏距离平方判定：$dx^2 + dy^2 \le R^2$。
    * 若在半径内，且当前指针不是 `VoidCell` 或边界，**强制调用该实体的析构逻辑**，并重新绑定一个干净的 `VoidCell`。
* **Benchmark 价值**：一个 $R=3$ 的黑洞，每帧最多会强制析构约 29 个多态对象并分配 29 个 `VoidCell`。随着它在屏幕上高速弹射，它会不断切断墙体脉络、吞噬流体、打断正在进行的化学反应，将高度有序的局部内存状态强行重置，极大地考验各类语言垃圾回收器（GC）处理短生命周期小对象爆发的吞吐量，以及 C++ 内存分配器的锁竞争/碎片合并能力。
